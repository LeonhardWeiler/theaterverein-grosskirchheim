1 · Offene Findings

LINT-1mittel eslint.config.js

Lint-Gate durch fehlenden Ignore unbrauchbar. eslint . meldet 124 Errors — aber 123 davon stammen aus dem gebauten, minifizierten Bundle .publish/assets/index-*.js (der Deploy-Worktree liegt lokal vor). eslint.config.js ignoriert nur dist, nicht .publish. Es bleibt genau 1 echter Fehler im Quellcode (siehe HOOK-1). Effektiv verdeckt das Rauschen echte Probleme.

Empfehlung: .publish (und ggf. public/assets) in globalIgnores([...]) aufnehmen. Danach ist eslint . aussagekräftig.

DATA-1mittel generate-images:41

Bild-Pipeline löscht die Originale. generate-images erzeugt -big/-medium/-small.webp und löscht danach das Original per fs.unlinkSync(filePath). Die Originale liegen unter public/assets/ (per .gitignore ausgeschlossen) — es gibt also keine versionierte oder sonstige Sicherung der Vollauflösung. Ein Teil-Fehler in sharp oder ein späterer Re-Run kann Quellmaterial unwiederbringlich verlieren.

Empfehlung: Originale nicht löschen, sondern in einen separaten, gesicherten (idealerweise versionierten oder per Backup erfassten) Ordner verschieben; Löschen höchstens optional und erst nach verifiziertem Ergebnis.

CLEAN-1niedrig src/data/vorstellungen.json.bak · .gitignore

Generiertes Backup ist eingecheckt. src/data/vorstellungen.json.bak ist versioniert — es wird von get-image-count als Backup erzeugt und gehört nicht ins Repo.

Empfehlung: Datei entfernen (git rm) und *.bak in .gitignore aufnehmen.

DEPLOY-1niedrig deploy.mjs · generate-sitemap.js

Deploy-Robustheit & Sitemap-Aktualität. Zwei Punkte: (1) git -C .publish commit -m "Auto deploy" schlägt fehl, wenn sich am Build nichts geändert hat — der catch meldet dann fälschlich „Deployment failed", obwohl es nur „nothing to commit" ist. (2) generate-sitemap.js wird im Deploy nicht aufgerufen; die sitemap.xml kann bei neuen Vorstellungen veralten.

Empfehlung: „nothing to commit" abfangen (Diff prüfen bzw. Exit-Code tolerieren); Sitemap-Generierung in den Build-/Deploy-Flow einhängen (z. B. als prebuild-Script).

HOOK-1niedrig src/components/carousel/hooks/useCarouselImages.js:16

Abgeleiteter State über setState im Effect. Der einzige echte Lint-Fehler (react-hooks/set-state-in-effect): images wird in einem useEffect aus folder/imageCount gesetzt, obwohl es sich um reinen abgeleiteten Zustand handelt — das erzeugt einen zusätzlichen Render-Durchlauf.

Empfehlung: images per useMemo(() => …, [folder, imageCount]) ableiten statt per Effect zu setzen.

DOC-1niedrig README.md · (fehlende) CLAUDE.md

Dünne Doku für ein Setup mit vielen Sonderwegen. Das README hat 2 Sätze. Weder die Generatoren (generate-images, generate-cover, get-image-count — braucht jq) noch der Worktree-Deploy (.publish, git worktree add) sind dokumentiert. Eine CLAUDE.md fehlt.

Empfehlung: README um Dev-/Build-/Deploy-/Asset- Workflow ergänzen (inkl. jq-Abhängigkeit und Worktree- Setup); CLAUDE.md via create-claude-md anlegen.

2 · Was schon stark ist

GOODstark

Durchdachter, eigenständiger Toolchain: deploy.mjs prüft Branch und sauberen Working-Tree, baut frisch, deployt atomar in einen Git-Worktree und legt ein sinnvolles .htaccess an (SPA-Fallback + langlebige, immutable Asset-Caches, kurzes HTML- Caching). Bild-Pipeline erzeugt echte responsive Größen via sharp (kein Vorbild-Widerspruch wie bei reinen public-Assets). Sitemap wird korrekt aus den Daten erzeugt. Keine Secrets im Repo, dist/.publish sind gitignored, Build ist grün. React Compiler aktiv.

3 · Bewusst entschieden

SPA mit React Router statt SSG (für eine kleine Vereinsseite ok); Worktree-Deploy nach publish statt CI/CD-Pipeline (schlank, ohne Runner); Apache-Hosting (daher das generierte .htaccess). Diese Entscheidungen sind stimmig und keine Findings — der Hebel liegt bei Lint-Ignore (LINT-1) und der Original-Aufbewahrung (DATA-1).
